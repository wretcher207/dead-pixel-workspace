Three Approaches for the Nutty Netties Build

All use Next.js + Tailwind + Framer Motion + Stitch-first workflow. The difference is architecture.

A

App Router + Static Export Recommended

Next.js 14+ App Router, output: 'export' for static build. Modern RSC-ready patterns, but outputs portable static files. Easiest to demo or deploy. Matches where Next.js is heading.

Pros

  • Modern, portfolio-impressive architecture
  • Fully static — deploy to Netlify, Vercel, or anywhere
  • No server required for demo
  • Showcases current Next.js best practices

Cons

  • Slightly more complex mental model than Pages Router
  • Framer Motion needs 'use client' directives
B

Pages Router + SPA

Classic Next.js Pages Router. Single-page feel with all sections on /. More familiar if you prefer the older pattern. Still fully featured for this use case.

Pros

  • Simpler architecture you already know from DPD
  • Less boilerplate around client components
  • Framer Motion integrates without 'use client' overhead

Cons

  • Legacy pattern — less impressive in a portfolio context
  • No RSC benefits
C

App Router + Full Dynamic (No Static Export)

Full Next.js App Router with server components, dynamic rendering, and potential for future API routes. More powerful but overkill for a spec site — adds complexity with no benefit here.

Pros

  • Maximum flexibility
  • Easiest to add a CMS or API later

Cons

  • Requires a Node server to run
  • No benefit for a static portfolio spec site
  • More deployment complexity
